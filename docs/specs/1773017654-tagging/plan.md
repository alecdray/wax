# Tagging — Implementation Plan

## Approach

Build a `tags` module following the exact same structure as `review`: a `Service` struct with business logic, a `adapters/` subdirectory with HTTP handlers and templ templates, and DTOs that get embedded in `library.AlbumDTO`. Tags are user-scoped — each user owns their tag vocabulary, and `album_tags` is the join table.

**Key decisions:**

- **Tag entry point**: The ellipsis dropdown in each album row gains a "Tags" menu item (mirroring the existing "Notes" item). Applied tags are also shown as compact chips in a new `Tags` column in the albums table.
- **Tag chips in row**: Show up to 3 chips inline with a "+N" overflow indicator. Keeps the table readable while making tags visible at a glance.
- **Predefined groups bootstrap**: "Sound" and "Mood" groups are created lazily on the first tag action for a user (a `GetOrCreateTagGroup` pattern). No login hooks or migration seeds needed.
- **Tag input UX**: An Alpine.js combobox — type to filter existing tags, press Enter to create new ones, click X to remove. A plain comma-separated field would make reusing existing tags friction-heavy and contradict "intuitive."
- **Tag removal**: Removing a tag from an album deletes only the `album_tags` row. The tag itself (in the user's vocabulary) is preserved. There is no tag management UI in v1.
- **Tag filtering of the albums table**: Out of scope for v1.
- **Shared tags**: Fully user-scoped; no system-level vocabulary.

Alternatives considered:
- Seeding "Sound"/"Mood" groups on login or via migration: rejected because groups are user-owned rows; lazy creation on first use is simpler and doesn't require touching auth or migration data.
- Separate icon button instead of dropdown item: rejected to keep the album row clean; the ellipsis dropdown is already the established pattern for per-album actions.

## Implementation Steps

1. **Database migration** — Create and run the migration for `tag_groups`, `tags`, and `album_tags`. Run `task db/up` to apply and regenerate schema + sqlc.

2. **SQL queries** — Write queries in `db/queries/tags.sql`:
   - `GetOrCreateTagGroup(user_id, name) tag_groups`
   - `GetOrCreateTag(user_id, name, group_id?) tags`
   - `GetTagsByUserId(user_id) []tags` (for the autocomplete — returns all tags the user has ever created)
   - `GetAlbumTagsByAlbumIds(user_id, album_ids) []row{album_id, tag_id, tag.name, tag.group_id, group.name}`
   - `GetAlbumTagsByAlbumId(user_id, album_id) []row` (single-album version, used in modal)
   - `CreateAlbumTag(user_id, album_id, tag_id) album_tags`
   - `DeleteAlbumTag(user_id, album_id, tag_id)`
   Run `task build/sqlc` after.

3. **Tags service** — Implement `src/internal/tags/service.go`:
   - `TagGroupDTO { ID, Name string }`
   - `TagDTO { ID, Name string; Group *TagGroupDTO }`
   - `GetOrCreateDefaultGroups(ctx, userId)` — ensures "Sound" and "Mood" exist for the user
   - `GetUserTags(ctx, userId) []TagDTO` — all tags the user owns (for autocomplete)
   - `GetAlbumTagsByAlbumIds(ctx, userId, albumIds) map[albumId][]TagDTO`
   - `SetAlbumTags(ctx, userId, albumId string, tagInputs []TagInput) ([]TagDTO, error)` — receives the submitted tag list, resolves each by name (get-or-create), diffs against current `album_tags`, inserts new rows, deletes removed rows, all in a transaction

4. **Library DTO extension** — Add `Tags []tags.TagDTO` to `library.AlbumDTO` and update `NewAlbumDTOFromModel` signature. In `GetAlbumsInLibrary`, call `tags.Service.GetAlbumTagsByAlbumIds` and populate each DTO.

5. **Tags templ templates** — Implement `src/internal/tags/adapters/tags.templ`:
   - `AlbumTagsCell(album, isOobSwap bool)` — renders tag chips for an album row cell (OOB-swappable, ID `album-tags-{albumId}`)
   - `TagsModal(album, allUserTags, tagGroups)` — the modal with Alpine.js combobox
   - `CloseTagsModal()` — ForceCloseModal wrapper
   Run `task build/templ` after.

6. **Tags HTTP handler** — Implement `src/internal/tags/adapters/http.go`:
   - `GET /app/tags/album?albumId=X` — fetch album + user tags, render `TagsModal`
   - `POST /app/tags/album?albumId=X` — parse submitted tag list, call `service.SetAlbumTags`, render `CloseTagsModal()` + `AlbumTagsCell(album, true)`

7. **Dashboard template updates** — In `dashboard.templ`:
   - Add `Tags` column header to `AlbumsTable` thead
   - In `albumRow`, add `<td>@tagsAdapters.AlbumTagsCell(album, false)</td>`
   - In the ellipsis dropdown `<ul>`, add a Tags `<li>` button that triggers `GET /app/tags/album?albumId=...`
   - Add `GetAlbumTagsID(albumId)` helper (like `GetAlbumRatingID`)
   Run `task build/templ` after.

8. **Server wiring** — In `server.go`:
   - Add `tags *tags.Service` to `services` struct
   - Instantiate `tags.NewService(db)` in `NewServices`
   - Register `tagsHandler` routes on `appMux`

## Database Changes

New migration creates three tables:

```sql
CREATE TABLE tag_groups (
    id         TEXT PRIMARY KEY,
    user_id    TEXT NOT NULL REFERENCES users(id),
    name       TEXT NOT NULL,
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(user_id, name)
);

CREATE TABLE tags (
    id         TEXT PRIMARY KEY,
    user_id    TEXT NOT NULL REFERENCES users(id),
    name       TEXT NOT NULL,
    group_id   TEXT REFERENCES tag_groups(id) ON DELETE SET NULL,
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(user_id, name)
);

CREATE TABLE album_tags (
    id         TEXT PRIMARY KEY,
    user_id    TEXT NOT NULL REFERENCES users(id),
    album_id   TEXT NOT NULL REFERENCES albums(id),
    tag_id     TEXT NOT NULL REFERENCES tags(id) ON DELETE CASCADE,
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(user_id, album_id, tag_id)
);
```

The `ON DELETE CASCADE` on `album_tags.tag_id` ensures that if a tag is ever deleted, all its album associations go with it. The `ON DELETE SET NULL` on `tags.group_id` means deleting a group orphans its tags as ungrouped rather than cascade-deleting them.

Run with: `task db/create -- add_tagging`, then edit, then `task db/up`.

## Testing

**Manual verification checklist:**

- Open the album table; confirm a "Tags" column is present (empty for all albums initially).
- Click the ellipsis dropdown for any album; confirm a "Tags" menu item appears.
- Click "Tags" — confirm the modal opens with an empty tag input and no existing tags.
- Type a new tag name and press Enter — confirm it appears as a chip in the combobox.
- Save — confirm the modal closes and the tag chips appear in the album row.
- Re-open the modal — confirm the previously entered tags are pre-populated.
- Type a partial name of an existing tag — confirm it appears as a suggestion in the autocomplete.
- Remove a tag chip and save — confirm it disappears from the album row.
- Open tags for a second album, type the same tag name used on the first album — confirm it autocompletes without creating a duplicate in the `tags` table.
- Verify that "Sound" and "Mood" groups appear as group options in the combobox on first use.
- Assign a tag to a group (e.g., "jazz" → "Sound") and confirm it appears grouped in the modal.
- Refresh the page — confirm tags persist across page loads.

## Risks & Mitigations

- **N+1 in bulk fetch**: `GetAlbumTagsByAlbumIds` must use a SQL `IN (...)` clause, not a loop. Match the pattern of `GetAlbumArtistsByAlbumIds`. With SQLite there is a max 999 bind parameters — if the library ever exceeds ~900 albums, chunk the query. This is not a near-term concern but should be noted.
- **Alpine.js combobox complexity**: The tag input is the most complex UI component in the codebase so far. Use a simple approach: an `<input>` bound to Alpine data, a filtered dropdown of existing tags below it, and a `chips` array for selected tags. Submitting the form POSTs the chips array as repeated `tag[]` form values.
- **Lazy group bootstrap race**: If two concurrent requests both try to `GetOrCreateTagGroup` for "Sound", the `UNIQUE(user_id, name)` constraint prevents duplicates — use `INSERT OR IGNORE` and then `SELECT`.
- **Group assignment UX scope**: The initial modal should allow assigning a tag to a group but the scope can be simplified — a dropdown per chip, defaulting to "ungrouped." If this proves too complex, defer group assignment to v2 and only create ungrouped tags in v1.
- **Library service import cycle**: `library.AlbumDTO` will import `tags.TagDTO`. `tags.Service` must not import `library` to avoid a cycle. The current pattern (review is imported by library, review doesn't import library) is the model to follow.
