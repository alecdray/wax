# Carousel Multi-View — Implementation Plan

## Approach

Use **Option A: HTMX lazy loading**. A single new route `GET /app/library/dashboard/carousel?view=<name>` returns an HTML fragment that replaces the carousel section. The initial page load renders the "Recently Spun" view inline (no extra round-trip), and clicking any other tab fetches that view's content on demand.

This matches the `GetAlbumsTable` and `GetFeedsDropdown` patterns already in the codebase, keeps `GetDashboardPage` fast (no additional library queries on initial load), and requires no Alpine.js state beyond a simple active-tab tracker that is already handled by HTMX's `hx-target` + `hx-swap`.

**Views for v1:**
1. **Recently Spun** — existing `GetRecentlyPlayedAlbums` query (last 20 by `played_at DESC`)
2. **Recently Added** — new lightweight SQL query: albums the user has in their library, ordered by their earliest `user_releases.added_at DESC`, limited to 20
3. **Top Rated** — new lightweight SQL query: albums with a non-null rating for the user, ordered by `rating DESC`, limited to 20

**DTO unification:** Introduce a `CarouselAlbumDTO` (defined in `listeninghistory` or a shared location — see Files section) with only the fields the carousel item needs: `ID`, `SpotifyID`, `Title`, `Artists string`, `ImageURL`. Each view's service method maps its source data into this type. This avoids the `RecentAlbumDTO` vs `AlbumDTO` mismatch in the template.

**Active tab state:** Tabs are rendered as part of the carousel fragment returned from the carousel endpoint, with the active tab determined server-side from the `?view=` query param. No client-side state storage is needed; the "Recently Spun" tab is active by default.

Alternatives considered:
- Option B (Alpine.js in-page switching) rejected because it forces all three views' data to load on every dashboard page load, including the expensive `GetAlbumsInLibrary` call.
- Option C (hybrid) rejected as unnecessarily complex for a first implementation.

## Implementation Steps

1. **Add new SQL queries** — In `db/queries/track_plays.sql`, add two new named queries:
   - `GetRecentlyAddedAlbums`: joins `user_releases` → `releases` → `albums` → `album_artists` → `artists`, groups by `albums.id`, takes `MIN(user_releases.added_at)` as `first_added_at`, includes `artist_names` via the same correlated subquery pattern already used in `GetRecentlyPlayedAlbums`, orders by `first_added_at DESC`, limits to 20, filters by `user_id`.
   - `GetTopRatedAlbums`: joins `album_ratings` → `albums` → `album_artists` → `artists`, filters `rating IS NOT NULL AND user_id = ?`, includes `artist_names` correlated subquery, orders by `rating DESC`, limits to 20.
   - Run `task build/sqlc` to regenerate the sqlc layer.

2. **Add `CarouselAlbumDTO` and new service methods** — In `listeninghistory/service.go`:
   - Define `CarouselAlbumDTO { ID, SpotifyID, Title, Artists, ImageURL string }`.
   - Change `GetRecentlyPlayedAlbums` return type to `[]CarouselAlbumDTO`.
   - Add `GetRecentlyAddedAlbums(ctx, userID) ([]CarouselAlbumDTO, error)`.
   - Add `GetTopRatedAlbums(ctx, userID) ([]CarouselAlbumDTO, error)`.

3. **Update `GetDashboardPage` handler** — In `http.go`, update the call to `GetRecentlyPlayedAlbums` to use the new `CarouselAlbumDTO` return type; update `DashboardPageProps.RecentAlbums` field type accordingly.

4. **Refactor and extend carousel templates** — In `dashboard.templ`:
   - Define a `CarouselView` string type and constants: `CarouselViewRecentlyPlayed`, `CarouselViewRecentlyAdded`, `CarouselViewTopRated`.
   - Rename `RecentlyPlayedCarousel` → `carouselStrip(albums []listeninghistory.CarouselAlbumDTO)` (unexported inner strip component).
   - Add `CarouselTabBar(active CarouselView)` — renders three tab buttons, each with `hx-get="/app/library/dashboard/carousel?view=<name>"`, `hx-target="#carousel-section"`, `hx-swap="outerHTML"`. Active tab gets a `tab-active` class.
   - Add `CarouselSection(albums []listeninghistory.CarouselAlbumDTO, active CarouselView)` (exported, replaces old `RecentlyPlayedCarousel`) — renders `<div id="carousel-section">` containing `CarouselTabBar` + the strip content, or an empty/placeholder message if `albums` is empty.
   - Update `DashboardPage` to call `CarouselSection(props.RecentAlbums, CarouselViewRecentlyPlayed)`.
   - Run `task build/templ`.

5. **Add `GetCarousel` HTTP handler** — In `http.go`:
   - Parse `?view=` query param, default to `recently-played`.
   - Switch on view name to call the appropriate service method.
   - Render `CarouselSection(albums, activeView)`.

6. **Register route** — In `server.go`, add:
   ```
   appMux.Handle("GET /app/library/dashboard/carousel", httpx.HandlerFunc(libraryHandler.GetCarousel))
   ```

7. **Build and verify** — Run `task build` to confirm the whole project compiles cleanly.

## Database Changes

Two new read-only queries in `db/queries/track_plays.sql` (no schema migrations needed, no new tables):

```sql
-- name: GetRecentlyAddedAlbums :many
SELECT albums.*, MIN(user_releases.added_at) as first_added_at,
    COALESCE((
        SELECT GROUP_CONCAT(a.name, ', ')
        FROM (SELECT DISTINCT ar.id, ar.name FROM album_artists aa JOIN artists ar ON ar.id = aa.artist_id WHERE aa.album_id = albums.id) AS a
    ), '') as artist_names
FROM user_releases
JOIN releases ON releases.id = user_releases.release_id
JOIN albums ON albums.id = releases.album_id
WHERE user_releases.user_id = ?
GROUP BY albums.id
ORDER BY first_added_at DESC
LIMIT 20;

-- name: GetTopRatedAlbums :many
SELECT albums.*, album_ratings.rating,
    COALESCE((
        SELECT GROUP_CONCAT(a.name, ', ')
        FROM (SELECT DISTINCT ar.id, ar.name FROM album_artists aa JOIN artists ar ON ar.id = aa.artist_id WHERE aa.album_id = albums.id) AS a
    ), '') as artist_names
FROM album_ratings
JOIN albums ON albums.id = album_ratings.album_id
WHERE album_ratings.user_id = ?
  AND album_ratings.rating IS NOT NULL
ORDER BY album_ratings.rating DESC
LIMIT 20;
```

Run `task build/sqlc` after adding the queries.

## Testing

Manual verification checklist:
- Dashboard loads with "Recently Spun" tab active and existing carousel data visible.
- Clicking "Recently Added" tab fetches and renders albums ordered by when they were added to the library; tab indicator moves to "Recently Added".
- Clicking "Top Rated" tab fetches and renders albums ordered by rating descending; tab indicator moves to "Top Rated".
- Switching back to "Recently Spun" works correctly.
- A user with no plays/no library/no ratings sees an appropriate empty state on each tab (no JS errors, no broken layout).
- The `?view=` query param can be bookmarked: loading `/app/library/dashboard` always opens the default view; the carousel endpoint itself (`/app/library/dashboard/carousel?view=top-rated`) is only meant for HTMX fragment fetches.
- Layout: tab bar + carousel strip fits within the existing dashboard scroll area without breaking the fixed-height layout.

## Risks & Mitigations

| Risk | Mitigation |
|------|------------|
| `GetRecentlyAddedAlbums` returns albums with no `user_releases` rows for some users | Query uses `JOIN` not `LEFT JOIN` so users with no releases get an empty result — show empty state |
| `CarouselAlbumDTO` type change breaks callers of `GetRecentlyPlayedAlbums` | Only one caller (`GetDashboardPage`) — straightforward update in the same step |
| `first_added_at` for albums with multiple releases is the earliest release — may feel surprising | Acceptable for v1; can be changed to `MAX` later if UX feedback suggests "most recently acquired" is more intuitive |
| `GetTopRatedAlbums` returns no results for users who haven't rated anything | Show empty state with a prompt like "No rated albums yet" |
| sqlc type for `MIN(user_releases.added_at)` may be `interface{}` (same as `MAX(played_at)`) | Apply the same `parseInterfaceTime` helper already used in `GetRecentlyPlayedAlbums` |
| Tab HTMX requests replace the entire `#carousel-section` including the tab bar — this is intentional (active state is server-driven), but causes a small visual flash | Acceptable for v1; could be mitigated with `hx-swap="innerHTML"` and moving the tab bar outside the swap target if it becomes a UX concern |
