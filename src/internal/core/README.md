# Core Module

The `core` module provides foundational infrastructure and utilities used throughout the application. It establishes architectural patterns, handles cross-cutting concerns, and provides shared components for HTTP handling, database access, authentication, and background task management.

## Package Structure

> **Note:** Packages with the `x` suffix (e.g., `timex`, `sqlx`) are extension packages that provide utility or extended functionality for their underlying package.

### `app`
Application-level configuration and state management.
- Environment-based configuration loading
- Global application state management
- JWT authentication with cookie and header support

### `contextx`
Enhanced context wrapper that provides type-safe access to request-scoped values.
- Use `ContextX` instead of `context.Context` throughout the application
- Provides convenient access to request-scoped state

### `httpx`
HTTP utilities for routing, middleware, and error handling.
- Custom router with automatic context enrichment
- Composable middleware system
- Consistent error responses supporting JSON and HTML fragments

### `db`
Database connection management and query interface.
- Automatic schema migrations
- Transaction support
- Type-safe query generation
- Connection pooling

### `task`
Background task scheduling and execution.
- Manages both cron-scheduled and ad-hoc tasks
- Standardized task interface
- Task registration and lifecycle management

### `templates`
Shared Templ components for consistent UI patterns.
- Layout and structural components
- Reusable UI elements
- HTMX interaction utilities

### Utility Packages

**`cryptox`**: Cryptographic utilities (encryption, decryption)

**`timex`**: Time-related utilities and constants

**`sqlx`**: SQL type helpers and utilities

**`utils`**: General-purpose utility functions

## Key Concepts

### Application State (App)

The `App` struct holds global application state and is injected into every request context:

```go
app := app.NewApp(config)
ctx = ctx.WithApp(app)
```

Access configuration and JWT claims through the App instance:
```go
app, _ := ctx.App()
config := app.Config()
claims := app.Claims()
```

### Context Pattern (ContextX)

Always use `contextx.ContextX` instead of `context.Context`:

```go
ctx := contextx.NewContextX(r.Context())
userId, err := ctx.UserId()
app, err := ctx.App()
```

The router automatically injects App and Request ID into every request context.

### HTTP Request Flow

1. **Mux** receives request and wraps context with App and Request ID
2. **Middleware** pipeline executes (logging, JWT validation)
3. **Handler** processes request with enriched context
4. **Error Handling** returns JSON or HTML fragment based on request type

### Middleware Composition

Middleware is composable and applied in order:

```go
// Global middleware
mux := httpx.NewMux(app, httpx.RequestLoggingMiddleware)

// Per-route middleware
mux.HandleFunc("/protected", handler, httpx.JwtMiddleware(spotifyService, userService))
```

### Error Response Pattern

Use `httpx.HandleErrorResponse()` for consistent error handling:

```go
httpx.HandleErrorResponse(ctx, w, httpx.HandleErrorResponseProps{
    Status: http.StatusBadRequest,
    Err: err,
    Response: *httpx.NewErrorResponse().SetComponent(errorComponent),
})
```

Supports both JSON responses and HTML fragments for HTMX.

### Database Transactions

Use `WithTx()` for transactional operations:

```go
err := db.WithTx(func(tx *db.DB) error {
    // All queries here are wrapped in a transaction
    tx.Queries().CreateUser(...)
    tx.Queries().CreateProfile(...)
    return nil
})
```

Transactions automatically commit on success or rollback on error.

### Background Tasks

Tasks implement the `Task` interface with run logic, optional scheduling, and identification.

Register and execute tasks:

```go
taskManager := task.NewTaskManager(db, logger)
taskManager.RegisterCronTask(myTask)         // Scheduled task
taskManager.RegisterAdHocTask(oneTimeTask)   // On-demand task
taskManager.Start(ctx)
```

## Authentication Flow

1. User authenticates and receives JWT in cookie
2. `JwtMiddleware` validates JWT on protected routes
3. User ID extracted and injected into `ContextX`
4. Services retrieve user from context using `ctx.UserId()`

JWT supports both cookie-based (web) and header-based (API) authentication.

## Configuration

Environment variables are loaded from `.env` file (see `.env.template`).

Configuration is environment-aware with conditional requirements:
- **Local**: Uses defaults, allows insecure secrets
- **Dev/Prod**: Requires explicit configuration, enforces secure secrets

## Database

- Uses SQLite with automatic schema migrations
- Queries generated by `sqlc` from `.sql` files in `db/queries/`
- Migrations managed by `goose` in `db/migrations/`
- Generated models in `db/sqlc/` (do not edit manually)

## Templates

Templ components provide server-side rendered HTML:
- Layout components handle page structure
- Component files use `.templ` extension
- Run `templ generate` after modifying `.templ` files
- Generated `_templ.go` files are automatically created
